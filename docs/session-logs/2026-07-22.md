- The console Home button was hardcoded to `move {joints:[0,0,0,0,0,0]}` — it never
  used the home pose. New route **`POST /api/robot/home`** (`backend/server.py`)
  calls `move_joints(HOME_JOINTS_DEG)` (non-blocking; the single-threaded server
  keeps serving joint polling while the arm travels). Front-end `home()` now POSTs
  it; the confirm text no longer says "all six joints to 0°". Single source of truth
  shared with the scan homing.

### 6. Hardcoded J2 (shoulder) rail guard
- **Context:** with the arm lowered, **J2 sometimes swings into the rail it's
  mounted on.**
- `backend/robot_bridge.py`: `J2_INDEX=1`, `J2_MIN_DEG=-120.0`, `J2_MAX_DEG=115.0`.
  `move_joints` **refuses** any command whose J2 target is outside **[−120°, 115°]**
  (the arm does not move; logged as `J2 RAIL GUARD`). Verified on the mock: in-range
  passes, out-of-range (both directions) refused, home (J2=93.33) allowed.
- Front-end `JOINTS0` J2 `max` 120 → **115** so the slider/input clamp matches the
  hard backend limit.
- **LIMITATION (the crux — see §8):** this only guards **joint-space** commands (jog
  / home). A **Cartesian** move (MoveL/MoveJ to a pose) lets the **controller** pick
  the joint solution, so it can still route J2 into the rail *mid-path*. We only find
  out by watching the move — too late.

### 7. Jog-to-targets fixed
- **Symptom:** "jog to targets doesn't work." Root cause: the COMMAND field is bound
  to the same `j.v` that live telemetry overwrites, protected only by a **4-second
  global timer** (`_jointHold`). If the timer lapsed before you clicked Jog, a poll
  overwrote your typed targets with the live pose → Jog commanded the *current*
  position → no motion.
- Fix (`gui/Scan Cell Console.dc.html`): replaced the timer with a **per-joint
  `edited` hold** — an edited joint keeps its typed target and is **not** overwritten
  by telemetry until the next **Jog** clears the flag. `setJoint` sets `edited:true`;
  the poll merge respects it; `jogToTargets` clears all `edited` after sending (so
  rows resume tracking live telemetry as the arm travels).

## 8. The problem this session surfaced, and the plan: **do the IK ourselves**

**Problem.** All IK currently lives in the **ROKAE controller**: we send it Cartesian
poses (MoveL/MoveJ) and it chooses the joint configuration. Because we don't compute
the joint path, we **cannot know whether J2 (or any joint) will hit the mounting rail
along the trajectory** — only at the commanded endpoints (and even those are the
controller's choice). We wanted to add the J2 range at the start, but a joint-space
clamp can't protect a Cartesian scan path; we'd have to run the move and watch.

**Solution (agreed).** Move the IK **off the controller into our own stack** — ROS 2
+ **MoveIt** ("do the IK in Frank's brain"). Then **every command to the arm is joint
angles** (a planned joint trajectory), not a Cartesian point. Wins:
- **Rail avoidance on the whole path:** model the arm's **mounting rail as a collision
  object**; MoveIt plans around it, so J2 never enters the rail — verified **before**
  execution, not discovered during.
- **Accurate sim:** the sim already runs FK on joint angles; feeding it the *same*
  planned joints makes it match the real arm exactly.
- **Determinism:** plan-fully-then-execute (already the target architecture — see
  `docs/architecture.md`; `MovementDriver` just streams joints).

**How hard.** Moderate — and it's the **intended end-state**, not a detour. Scaffolding
already exists: `ros2_ws/src/qc_moveit_config/` (MoveIt config, `kinematics.yaml`,
`joint_limits.yaml`, `qc_move_group.launch.py`), `path_planner/moveit_planner.py`, and
the built `sr5_arm_driver` (which already exposes joint moves). Remaining work:
1. Stand up **MoveIt in Docker/Humble** (the host has no ROS 2) with the SR5 URDF +
   the **rail + mounting rail as collision geometry**.
2. Plan **joint trajectories** with collision checking (IK via KDL/TracIK/bio-ik);
   make the solution branch match the controller's kinematics.
3. Command the arm in **joint space** (`MovementDriver` → `move_joints` / joint
   trajectory), retiring Cartesian MoveL for scans.
4. Bridge **console ↔ ROS 2 over rosbridge** (planned architecture).
5. Drive the **sim from the planned joints**.
A first slice (one part, rail as a collision object, joint-trajectory execution) is a
few focused sessions; full parity (all primitives, calibration, robustness) is more.

**Interim mitigation in place now:** the hardcoded **J2 guard (§6)** protects
joint-space commands (jog/home). For Cartesian scans, the driver already pre-flights
each waypoint with `calcIk`; a follow-up could **reject a waypoint whose IK solution
puts J2 out of [−120, 115]** — but that's still endpoint-only and branch-dependent,
which is exactly why owning the IK is the real fix.

## 10. Fraction 0.00 → 1.00 — the scan trajectory now plans (gap in §9 CLOSED)

Diagnosed the fraction-0 precisely (headless IK probe): the part sat at its CAD
origin — straddling the base singular column (x=y=0) and mostly below reach (z as
low as 0.008 m; the flange floor is ~0.09 m above the table) — so only 5/12 sampled
waypoints had IK, and the cartesian lines broke. Two fixes:

1. **Reachable placement** (`path_planner_node._reachable_shift`): shift the part so
   its footprint centre sits at `planner.scan_anchor_m` (default [0.30, 0.0], off the
   singular column) and its lowest point is raised to `planner.pedestal_z_m`
   (default 0.40 m, the reach sweet-spot). Placeholder for the measured corner
   transform + raised fixture (decision 5 / §2). → IK **5/12 → 12/12**.
2. **IK-seeded cartesian start_state** (`moveit_planner.compute_ik` + `/compute_ik`
   client): each raster line's `compute_cartesian_path` now starts AT the line's
   first waypoint instead of the robot's current straight-down pose (the real cause
   of fraction 0 — the first straight-line segment from the down pose was infeasible).

Result on the smallest part: **min cartesian fraction 1.00**, `/plan/trajectory` =
**1139 collision-free joint points** across 9 lines. Config knobs documented in
`config/system_config.yaml` (planner.scan_anchor_m / pedestal_z_m).

Still TODO (further quality, not blockers): free-space (OMPL) moves BETWEEN lines,
time-parameterisation, and replacing the placeholder placement with the measured

## 11. Safety + preview follow-ups (5 cm buffer, gantry display, jog check, free-space)

Triggered by: the arm self-collided when the operator JOGGED a joint. Key insight —
**jog bypasses MoveIt entirely**, so no planning buffer prevents it; the fix is a
collision-checked jog.

- **Gantry in the console sim** — `viewer3d.js` now draws the plate + 2 posts
  (mirrors the URDF collision geometry) so the operator SEES what J2 can hit.
- **"5 cm buffer around parts"** — implemented as MoveIt robot link_padding
  (`moveit_planner`, `QC_COLLISION_PADDING_M`), but a blanket 5 cm is PHYSICALLY
  INCOMPATIBLE here: the SR5 is bolted ~1 cm below the mounting plate, so a 5 cm pad
  makes its own shoulder collide with the plate and folded scan poses self-collide —
  measured fraction 1.00 → 0.14. Default reverted to 0 (true geometry is already
  collision-free; the scan clears the part by the 250 mm standoff). Tunable if a
  small margin is wanted.
- **Collision-checked jog** (the real fix) — the console pre-flights each jog target
  against MoveIt `/check_state_validity` over rosbridge and refuses self/gantry
  collisions (`ros_bridge.js` gained `callService`; console `_checkCollision` +
  `jogToTargets` guard; fail-open with a warning if the ROS graph is down). MECHANISM
  PROVEN headless over rosbridge: J2=1.1 rad → invalid, contact link4↔gantry_post;
  zero pose → valid. In-browser JS wiring still needs a browser to confirm.
- **Free-space between lines** — `moveit_planner.plan_freespace` (OMPL via
  `/plan_kinematic_path`) + bridge insertion in `plan_scanpath`. CODED but NOT YET
