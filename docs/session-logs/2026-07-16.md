## The debugging arc (each step taught the next)

1. **Frame math was fine.** Proved the typed base-frame mm round-tripped to the
   controller unchanged. Not a units/frame corruption bug.
2. **Wrist singularity (wrong lead).** "Go to point" hard-coded orientation to
   rpy(0,0,0) = tool straight down = the wrist singularity at home (J5=0). Fixed
   by holding the CURRENT orientation (`keep_orientation`) instead. Didn't fully
   fix it.
3. **We were flying blind.** `moveStart` returns as soon as the command is
   QUEUED; the controller resolves IK / checks limits AFTER and reports failures
   via the **`moveExecution` event**, which we never read (and the shared `ec`
   dict was never checked). Added full instrumentation: every backend line now
   prints to the server terminal (`_blog` prints + ring buffer); dumped current
   pose, target pose (trans/rpy/**confData**/elbow), both `ec` dicts, the full
   `moveExecution` event, and a decoded error table.
4. **Real errors, in sequence:** 50002 (MoveL straight-line singularity) → switched
   go-to to MoveJ → 50023 (MoveJ empty confData → singular branch) → seeded
   confData from current pose + `setDefaultConfOpt(True)` → 50021 (no solution for
   forced conf) AND the arm silently didn't move.
5. **The `confOpt(True)` mistake.** Forcing conf made the controller ACCEPT the
   command but not execute. Reverted to `setDefaultConfOpt(False)` with confData
   as a SOFT bias (Ra's working debugger's recipe). Added `operationState`-based
   motion confirmation (moving→idle, "didn't start = rejected") — the missing
   check that finally told us the truth.
6. **THE FIX — frame.** `calcIk` returned **-32 for every target**. Root cause:
   we were reading/commanding in **`flangeInBase`**, but `calcIk` (and the
   controller's reachability) resolve against the reference/work-object frame,
   **`endInRef`** — the frame Ra's working xCore debugger uses throughout. Switched
   the whole Cartesian path to `endInRef` + added a `calcIk`/`checkPath` PRE-FLIGHT
   (no motion) that rejects singular/unreachable targets before dispatch. **Go-to
   then worked on the real arm.**

## The working recipe (what makes real-arm Cartesian motion go)

- Read/command in **`endInRef`** (NOT flangeInBase). Operator types controller-frame
  mm matching the readout; no table↔base conversion for go-to.
- **Pre-flight before any motion:** `robot.model().calcIk(target, Toolset(), ec)`
  (endpoint reachable?) + `robot.checkPath(start_pose, start_joints, target, ec)`
  (straight-line MoveL clean?). Reject with the decoded reason; send nothing.
- `setDefaultConfOpt(False)` + copy current pose's `confData`/`elbow` onto the
  target as a SOFT IK bias. NEVER force `confOpt(True)` (accept-but-don't-move).
- Pass `zone` to the command. MoveJ = point-to-point (endpoint only). MoveL =
  straight line (needs a clean path; stricter).
- Confirm motion via **`operationState`** (moving→idle); "didn't start within 3 s
  and no error" = silent refusal.
- The controller assigns its OWN path IDs (`j#N`/`l#N`) and ignores the string we
  pass — do NOT match on our cmdID.
- Errors come through the `moveExecution` event / pendant, NOT `moveStart`'s ec.

## Error-code cheat sheet (observed)

- **50002** — MoveL target/path is a Cartesian singularity.
- **50023** — MoveJ target picked a singular IK branch (confData empty/incorrect).
- **50021** — MoveJ target has no solution in the specified (forced) confData.
- **-32** — `calcIk` no solution (wrong frame, or genuinely unreachable/singular).
- **-50002 / -50102** — `checkPath`: straight-line unreachable / singularity.
- **-17** — `moveStart` with motors OFF ("motor power state unsupported operation").

## Continued — path tracing works end to end (later same day)

After go-to worked, we extended the same machinery to the whole system:

1. **Orientation search + MoveJ fallback** — `calcIk` checks a POSE (position +
   orientation); a position reachable at one tool angle often isn't at another, so
   holding a fixed angle false-fails far targets. `move_pose(search_orientation=True)`
   sweeps orientations to find one that reaches the POSITION ("just get the head
   there"); `allow_joint_fallback=True` drops MoveL→MoveJ when the straight path is
   singular but the endpoint is reachable. Go-to uses both → reaches any point in
   the feasible region.
2. **calcIk must run on the BARE target** (no confData) — seeding the current
   pose's confData onto a far target made calcIk demand that exact config and
   return -32 for a reachable point. confData is copied only onto the MOVE (soft
   bias; confOpt False ignores it anyway).
3. **`follow_path(position_only=True)`** — traces waypoint POSITIONS via the go-to
   machinery (orientation searched per point). Wired into all three send paths:
   debug-viewport "Send to arm", Debug-tab "Send to robot", Run-flow "Confirm &
   start scan".
4. **THE path-frame fix — self-calibrating flangeInBase→endInRef.** The controller
   COMMANDS in `endInRef` (negative z), but our table→arm math yields `flangeInBase`
   (positive z) — so every path waypoint was sent in the wrong frame and rejected
   ("nothing moves"). `endInRef != flangeInBase` (identity toolset notwithstanding;
   they differ by the base↔ref roll). Fix: at connect, read the current pose in
   BOTH frames and derive the fixed rigid transform (`_solve_frame_transform`);
   `follow_path` now does table → flangeInBase → **endInRef** before commanding.
   go-to is unaffected (operator types endInRef directly). Note: `setAvoidSingularity`
   IS available but only on `xMateRobot` (not `Cobot_6`); opt-in via
   `QC_ROBOT_CLASS=xMateRobot QC_AVOID_SINGULARITY=1` — off by default (wrist method
   drifts orientation, bad for scan precision).

**Debug 3D viewport (`gui/viewer/debug.js` + `debug.html`, `viewer3d.js`):**
- Shape trace is now **position-only** (tip follows the wireframe; orientation

## Continued (2) — oriented continuous scan → X-raster (committed after `0a7c814`)

Built the real scan-path system on top of the working move machinery. Files:
`backend/robot_bridge.py`, `backend/server.py`, `gui/viewer/{debug.js,debug.html,viewer3d.js}`.

1. **Oriented continuous scan** (`scan_trace` / `start_scan_trace`, endpoint
   `/api/robot/scan_trace`): each scan line is one CONTINUOUS blended MoveL sweep
   (`move_pose_list` = append all + moveStart once, turning zone) with a MoveJ
   reposition between lines. Per-waypoint reachability via `calc_ik_ok` + a **±10°
   incidence cone search** (`_cone_reachable`); **unreachable waypoints are skipped,
   not fatal** (down-facing/below-table points drop out).
2. **Self-calibrating flangeInBase→endInRef** (`_solve_frame_transform`, read at
   connect from the pose in both frames): the controller commands in `endInRef`
   but our table math yields `flangeInBase` (they differ ~180° — z sign flips).
   Without this every path waypoint was the wrong frame → unreachable → nothing moved.
3. **Registration saga → FIXED ORIGIN.** First anchored the part at `home ± 0.15z`
   but (a) the sign was backwards (`-0.15` pushed it past reach; `+0.15` is the
   reachable side) and (b) reading the *live* pose as "home" made the part DRIFT as
   the arm moved. Resolved per Ra: **part centred at the fixed origin (0,0,0 = arm
   base, table frame)**, no live-pose read, no drift.
4. **THE PATH = overhead X-raster** (`_build_scan_poses`): a regular grid over the
   part's XY footprint, sweep +X / step Y / sweep −X (boustrophedon), scanner
   looking straight DOWN from `STANDOFF_M` (250 mm) above the part top. Surface-
   normal aiming was abandoned for the clean raster (normals on a 3-D part scatter
   the standoff positions into a 600 mm shell). Tunable: `QC_RASTER_STEP_M` (grid),
   `QC_STANDOFF_M`. Preview: `/api/robot/scan_preview` returns the reachable poses
   in the viewer frame (no motion).
5. **Debug viewport rebuilt** for the CAD-scan flow: removed the shape-outline
   feature (its Trace/Send buttons confused things). Now: **90° flip buttons**
   (Flip X/Y/Z, part re-grounds on the table) → **Generate path** (client draw +
   backend reachability, drops below-table points so nothing clips the floor) →
   **Scan → arm**. Camera-follow is always on. Part-orient gizmo/sliders were tried
   then replaced by the flip buttons.
